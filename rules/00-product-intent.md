# Product intent

## What NU:TONIC is

- A **multi-user geo-guessing game**: players infer real-world locations from fragmented cues (memory / satellite / distorted map data). **Non-ranked** play uses **per-`map_id` leaderboards stored on the device**—scores are **not** submitted to servers by default (`rules/05-networking-leaderboard.md`). **Cross-player, server-visible** async competition (same published **`map_id`**, no lobbies) applies to **ranked** missions and/or any **optional** community sync the product documents in OpenAPI (`docs/SOCIAL-AND-COMPETITION.md`). **Human / Astronaut / Alien** are **roles** that **tag** local rows (and optional server rows) and **enrich** filters—not mandatory team queues.
- **Roles are player-selected and first-class:** every player explicitly chooses **Human**, **Astronaut**, or **Alien** (see role selection screen and `rules/01-navigation-architecture.md`). UI must show the same three roles consistently (icons + short copy); **mechanics, assist tuning, levels, scoring formulas, and fairness for normal / casual play live in `commonMain`** so Android, iOS, Desktop, and Web stay aligned. **Ranked missions** move **round truth and score math** to the **server** for that session (`docs/RANKED-MODE.md`). **Server-assisted** ML (**TerraMind TiM**, optional **`terramind_v1_*_generate`**, VLM, TerraMesh-backed caches) is an enhancement, not a substitute for **ranked** verification rules.
- **Narrative + single guess loop:** Missions are **story-forward** using **simple text boxes** on **mission selection**, **map / level selection**, and **on-map briefing**—backed by **Markdown in `prompts/`**, **serialized at build time** (see `docs/NARRATIVE-AND-PROMPTS.md`). Players **submit one primary guess** per round (no mandatory chain of “repeat rounds” before submit). Optional pin adjust **before** confirm is fine.
- **Play timer (cosmetic only):** During **`PLAY`**, the client **may** show **elapsed time counting up** against a notional **`play_budget_ms`** from the bundled mission / map catalog—**diegetic HUD and INTEL session copy only**; it is **not** a score input, **not** verified, and **must never** cause a fail, phase lock-out, or submit rejection (**non-ranked and ranked**). **Pause while backgrounded** (do not advance elapsed) is **client-only** polish. The server **does not** track or enforce play duration on **start** or **submit** (`docs/GAME-ENGINE.md` §7.3, `docs/RANKED-MODE.md` §4).
- **Map UX:** Players navigate a **world map** with **satellite / map / hybrid** basemap where the platform supports it, plus a **reference Mapbox (or equivalent) still**; **search and submit** live in an **expandable bottom-right modal** tied to that reference (`rules/04-maps-and-gameplay.md`).
- **SCAN play:** Rounds ship **cached Mapbox downscaled stills** as the **primary** geographic reference, plus **`data/`** / **Dataset** catalogs and **optional pre-cached assists**: **AI Street View descriptions** (batch LFM-VL) and **multi-tier useful hints** (default **six** monotonic bands; **no coordinate literals in hint text**, including the strongest tier)—see `docs/GAME-ENGINE.md` §9 and `docs/NARRATIVE-AND-PROMPTS.md`. **No lobbies** and **no wait** for other players; optional **peer marker** is a **hint** only. **Non-ranked:** assists optional, **no** score penalty. **Ranked:** assists **forfeit** verified placement if used before `submit` (`docs/RANKED-MODE.md`). The **narrative overlay** stays **always invocable** for **authorial** `prompts/` and **user-typed** input (tone per `docs/DESIGN.md`); assists use **separate collapsible** UI. **`On-device` inference is required for the PRO tab only** (port of **`refs/VLMExample/`** per `rules/06-server-vlm-tim-and-on-device-ml.md`, `docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md`).
- **Game-first UX**: low cognitive load, **few taps to act**, clear feedback loops. Not a utility app.

## Success criteria for any implementation

1. **Parity**: The same **routes**, **game loop**, and **data shown** on Android, iOS, Desktop, and (if in scope) Web—differences are only where platform APIs require it (e.g. map provider, GPS for **optional** POI submit vs manual/map pick). **Game server session:** clients **authenticate to the game server** (device/session token or JWT—**no full account required**) on startup or first API use so the server can **rate-limit**, **cache expensive responses**, and reduce leaderboard / hydration **spam** (`rules/05-networking-leaderboard.md`). This is **not** the same as optional **social accounts**. **Leaderboard** views are **per-`map_id`** wherever the UI is map-scoped.
2. **Client authority (gameplay)**: **Round/match state, mission/assist configuration, scoring, XP/levels, and fairness** are **owned by the client** (`commonMain` state machine) for **non-ranked** missions whose truth is **local or bundle-defined**. **Ranked missions** are the explicit exception: **server** holds round truth until resolve and **recomputes** scores (`docs/GAME-ENGINE.md` §0, `docs/RANKED-MODE.md`). **Non-ranked** leaderboard rows are **persisted locally** per `map_id` (see `rules/13-client-cache-and-data-plane.md`); **optional** community **`POST`** is not part of the default product path.
3. **Lightweight client + POI assets**: **Ranked and non-ranked levels** draw from datasets under **`data/`** (and derived bundles), enhanced by **build / CI scripts** that hydrate narrative, manifests, and **pre-cached Mapbox clue stills**—see `docs/NARRATIVE-AND-PROMPTS.md`, `docs/GAME-ENGINE.md`, `plans/2026-04-07-complete-implementation-architecture.md` §2. **POI proposals** go **only** through the **game server** (`POST .../poi`); the server persists and republishes them for **map selection**, **bundle refresh**, and **ranked** eligibility once accepted—**no client-only path** into the catalog (`docs/POI-PACKAGES-AND-OFFICIAL-CLIENTS.md`, `rules/05-networking-leaderboard.md`).
4. **On-device inference (PRO only):** The **PRO** tab **must ship** the on-device vision–language path as a **behavioral port** of **`refs/VLMExample/`** (inference API: `ModelRunner`, multi-image messages, caption + JSON boxes—see **`docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md` §6**, **`rules/06-server-vlm-tim-and-on-device-ml.md`**). **VLM model weights and runtime bundles** are **part of the client build/publish pipeline** (packaged assets + CI/store alignment to `model_bundle_id`)—not ad-hoc Hub pulls on device. **SCAN** does **not** require on-device VLM—bundled stills suffice. **`POST`** bodies follow **OpenAPI** (`docs/POI-PACKAGES-AND-OFFICIAL-CLIENTS.md` §4).
5. **Marketplace / “official app” posture (store builds)**: When product distributes via **app stores**, any **server-mutating** **score** or **POI POST** (community sync, ranked, catalog writes) should require **short-lived JWTs**, **registered client identities** (package + signing cert / App Attest where used), and **strict OpenAPI schemas** (`rules/05-networking-leaderboard.md`, `docs/POI-PACKAGES-AND-OFFICIAL-CLIENTS.md`). **Pure local** non-ranked leaderboards **do not** hit authenticated score endpoints.
6. **Ranked missions**: When a mission is **ranked**, **server-held** secrets for active rounds, **guess-only** submits, **server-recomputed** distance/score, **`round_ticket`** (or equivalent) binding, and **separate** ranked aggregates from non-ranked boards apply (`docs/RANKED-MODE.md`, `rules/05-networking-leaderboard.md`). **User POI** accepted by the server is **immediately eligible** to appear in **map selection** and **ranked** round pools for other players—server processing is the **only** gate; never trust a client-only flag (`ranked_eligible`, etc.).
7. **Server role (optional reference implementation)**: A Python/FastAPI (or similar) server may **optionally aggregate** community score submissions (when shipped), serves **versioned POI bundles** (full or **redacted** per `round_type` / tier), optional VLM and **TerraMind TiM / generation**-related endpoints (when shipped), and **ops** UI (`/ops`). **Trust for score math** follows **ranked** vs **local non-ranked** vs **optional** community rows (`rules/05-networking-leaderboard.md`); store builds add **JWT / official client** for **documented** write paths. See `rules/06-server-vlm-tim-and-on-device-ml.md`, `rules/13-client-cache-and-data-plane.md`.
8. **Design fidelity**: Screens match the **structure and hierarchy** of `refs/stitch` mocks and the **tokens and rules** in **`docs/DESIGN.md`** with **`rules/02-design-system.md`** for how they land in Compose (glass surfaces, glow discipline, **Space Grotesk + Inter + Orbitron** roles, no runtime font CDN).

## Out of scope for “pixel-perfect HTML port”

- Do not ship the Tailwind/HTML as the app. **Compose** (and native map/embed where needed) is the implementation; HTML is **reference only**.

For a full technical breakdown of vendoring HTML, WebView hybrids, and how that relates to `kotlin-js-store` / Kotlin/JS builds, see **`09-html-vendoring-and-interface-stack.md`**. Deviations require an explicit product/ADR decision.
