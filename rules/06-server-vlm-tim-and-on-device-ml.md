# Server-side TerraMind **TiM**, TerraMind **generation**, VLM, and on-device ML

This file describes **server-side TerraMind** in two allowed forms—**Thinking-in-Modalities (TiM)** via **`BACKBONE_REGISTRY`** `*_tim` models, and **full multi-modal generation** via **`FULL_MODEL_REGISTRY`** `*_generate` models ([TerraMind guide — Generation](https://terrastackai.github.io/terratorch/stable/guide/terramind/))—plus **SCAN** narrative overlay (authorial + user text; **location clue** = **cached Mapbox still** from scripts/Datasets per **`docs/GAME-ENGINE.md` §9**), and **PRO**: **coordinate-driven** server materialization (Mapbox + Sentinel-2, downsized for contracts) **merged** with optional **TiM** / **`_generate`** summaries on the **game server plane**, **alongside** **on-device** multi-image VLM (per **`refs/VLMExample/`** and **`docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md`**), and how they relate to **client-authoritative** non-ranked gameplay (`rules/00-product-intent.md`, `rules/04-maps-and-gameplay.md`).

**Scope note:** **TiM** and **`terramind_v1_*_generate`** are **in scope** for **`inference/*`** and worker **HTTP APIs**, **HF Jobs**, and **standalone Gradio demos**—**not** in-process on the thin **`server/`** game API. Generic **non-TiM, non-generate** TerraMind **backbone-only** encoders (e.g. `terramind_v1_base` without `_tim` / `_generate`) remain **out of scope** unless an ADR adds them—prefer **`_*_tim`** or **`_*_generate`** for server EO so modality contracts stay explicit.

## Rule: TiM and TerraMind generation stay off the client

- **TiM** and **`_generate`** inference (TerraTorch; registries per the [TerraMind guide](https://terrastackai.github.io/terratorch/stable/guide/terramind/)) run **only in Python processes that load TerraTorch**—**not** in Kotlin clients and **not** in the **thin** **`server/`** game API process (`plans/2026-04-07-game-server-thin-orchestrator.md`, `docs/SERVER-AND-INFERENCE-ARCHITECTURE.md`). Concretely: **HF Jobs** (batch → Dataset), a **TerraMind worker** Space or container the game server calls over HTTP, **`inference/*`** services and **`demos/terramind_space/`** (see **`12-python-gradio-terramind-server.md`**, **`plans/2026-04-07-gradio-terramind-backend.md` §2**)—same stack, **different deployable** than thin orchestration.
- **TerraMesh** in product use is **narrow**: **cached AI-guess lat/long** (and related metadata) **per `map_id`**, produced server- or job-side and consumed over HTTP (`rules/10-terramesh-vlm-progressive-zoom-game-engine.md`). **Hydration source (normative):** the same **TerraMind TiM** worker the game server calls must expose **all configured `tim_modalities` outputs** (per-modality, schema-capped JSON — see `docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md` **`tim_modality_outputs`**). The **`Coordinates`** imagined modality, when enabled and present in the forward, **feeds `ai_lat` / `ai_lon`** (and confidence) for **`AiGuessStore`** / Dataset rows used on the **`AI_GUESS`** path (`docs/GAME-ENGINE.md` §12.2) alongside legacy TerraMesh-only batches — **scoped to `map_id` / clue materialization**, not arbitrary user PRO pins unless OpenAPI defines an explicit registration flag. **PRO** tab bundles merge the **same** per-modality object for the dashboard strip. Clients do not embed PyTorch or TerraMesh loaders.

**Implications (read before wiring persistence):** **`AiGuessStore`** answers “what **fixed** AI guess does **this published round** ship?” A **PRO** TiM forward answers “what does the model think at **this ad-hoc pin**?” Default **PRO** persistence is **job / bundle** only. Piping every PRO **`Coordinates`** into **`AiGuessStore`** (a) **mixes** analyst pins with curated **`map_id`** rows, (b) can **bypass** **`POST .../poi`** and moderation for content that surfaces on maps or ranked pools, and (c) inflates **retention and `content_version` churn**. **Normative detail:** `docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md` **§1.1.1**.

## Rule: SCAN narrative + optional assists (Mapbox primary; assists labeled)

- The **map narrative overlay** (authorial + user text) must be **always invocable** during SCAN play (glass overlay per **`docs/DESIGN.md`** and **`rules/08-ux-and-performance-footguns.md`**).
- **Primary geographic reference** for matching remains the **downsampled Mapbox still** from **script- / Dataset-built** bundles (`docs/GAME-ENGINE.md` §9). **Optional assists**—**pre-cached Street View description text** (batch LFM-VL), **pre-cached useful hints** (three tiers: continent → regional EO features → country), and **optional peer marker** after **Reveal uplink**—live in **separate collapsible** panels labeled as assists, **not** inside authorial copy. **Non-ranked:** assists do not change score math. **Ranked:** using these assists before `submit` **forfeits** verified placement (`docs/RANKED-MODE.md`, `rules/05-networking-leaderboard.md`). Do not block map pan or the guess modal (`rules/08-ux-and-performance-footguns.md`).
- **Optional** manifest refresh may ship **new `content_version`** slices between releases; default play is **fully offline** from bundled assets (`rules/13-client-cache-and-data-plane.md`).

## Rule: on-device ML is mandatory for the PRO tab only (VLMExample port)

- **PRO** is a **coordinate info dashboard**: user **lat/lon** → **game server** → **standalone PRO materialization service** (Mapbox + Sentinel-2, **downsampled** to the **on-device VLM** multi-image contract and to **TiM** / **`_generate`** input rules) → optional **TerraMind worker** → **`ProVisionBundle`** returned with **`tim_modality_outputs`** (all **`tim_modalities`** the run configured — logits, thumbs, **`Coordinates`**, etc., each schema-capped) plus **`terra_mind_generate_summary`** when applicable; the client runs **on-device** VLM over **`vlm_image_set`** for **caption + labeled bboxes** and **composes** a **layered** card (imagery + boxes + caption + server strip)—see **`docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md`**. **SCAN** does **not** require on-device VLM.
- **On-device VLM weights ship with the product build/publish pipeline:** checkpoints and engine files needed at runtime are **packaged with the client** (Gradle/assets/`composeResources` per target), **checked in CI** (presence, size, **`model_bundle_id`** alignment with materialization), and submitted with **store / distribution** artifacts—not pulled ad hoc from Hub or unknown URLs in production (`docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md` §6.0, `rules/13-client-cache-and-data-plane.md`).
- **On-device inference** is implemented as a **behavioral port** of **`refs/VLMExample/`** (`ModelRunner`, multi-image **`ChatMessage`**, caption + JSON boxes). **`expect`/`actual`** bridges non-Android targets until parity; optional **delta** model updates stay **OpenAPI-pinned** if product enables them.
- **PRO** **must implement** the on-device vision–language path (architecture, loading, inference hooks) per **`refs/VLMExample/`** and that spec on targets that ship **PRO**. Product may set **`pro_inference_route`** so **`_generate`**-derived copy is **primary** on the dashboard while the device still runs VLM for boxes/caption, or **hybrid**—OpenAPI defines enums.
- **Inject** only **allowlisted** context into local PRO models; **cap tokens** and **CPU/GPU** budget; **never** block **guess submit** or **lock-in** paths on SCAN.
- **Ranked missions:** Keep **PRO** tab UX disjoint from **active ranked SCAN** sessions so operators do not mix job state (`docs/RANKED-MODE.md`, `docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md`).
- **Outputs** on **`POST`** paths follow **OpenAPI** field lists (`rules/05-networking-leaderboard.md`).
- **PRO tab:** **Compose** surface built on the **VLMExample** inference port, not a raw chat clone—**initial build:** UI is **gate-aware**; the game server **entitlement check always returns allowed** for **valid registered clients**—see **`docs/PRO-TAB-VLM-ORCHESTRATION-SPEC.md`** §3.1 / §9 (`rules/01-navigation-architecture.md`).

## Rule: lightweight client + POI packages

- **Primary clue assets** for published rounds should come from **server-built or bundled POI packages** (e.g. downsampled `mapbox/*.png` + metadata from `poi.json`—see **`docs/POI-PACKAGES-AND-OFFICIAL-CLIENTS.md`**), not from ad hoc network scraping on device.
- **Cached rounds:** prefer **precomputed** manifests; local models **may** enrich UX offline but must **fail soft** to static copy.

## Alien / Human / Astronaut roles

- **Role selection** is **always** explicit in the UI (**Human**, **Astronaut**, **Alien**). Per **`docs/NARRATIVE-AND-PROMPTS.md`** (lore and roles), **Human / Astronaut / Alien** are **narrative and presentation only**: they do **not** change scoring, ranked fairness, baseline clue difficulty, or core hint mechanics unless an ADR adds explicit role-gated mechanics and documents them in OpenAPI.
- **Generated or authored copy** may still **address the player in character** (e.g. salutation by role) when templates allow—**presentation only**, same win/loss math for all roles.
- **“Unpredictable AI playstyle”** in marketing refers to **AI opponents / synthetic markers** (heuristics, **per-`map_id` TerraMesh cache**, or **TiM**-backed **`AiGuessStore`** rows where product allows), **not** to giving the **Alien player** different rules than Human or Astronaut. **Do not** ship Alien-only scoring or hint logic without product sign-off and cross-platform parity.

## Latency and resilience

- **Timeout and fallback**: If **TiM**, **generation**, VLM, or on-device inference fails, degrade with **cached** clues, **static** copy, or **pre-hydrated** outputs—**never** hang the map UI or block the **guess modal** submit path.
- **AI marker phase:** The **client engine** must still reach a resolved round with an **AI guess coordinate** (heuristic, **per-`map_id` TerraMesh cache**, bundled table, or optional **server TiM / generation / VLM** response) for UI parity. Document fault paths if a round aborts (`docs/GAME-ENGINE.md`).

## Relation to reference code under `refs/`

- Use **`refs/`** Python or other reference **as behavioral reference** for **server or batch** jobs. Kotlin clients consume **documented HTTP (REST)** APIs; do not copy Python dependencies into KMP.

## TerraMind — **TiM** (`BACKBONE_REGISTRY`, `*_tim`)

- Use **TerraTorch** **`BACKBONE_REGISTRY`** with **`*_tim`** model names and **`tim_modalities`** exactly as documented in the [TerraMind guide](https://terrastackai.github.io/terratorch/stable/guide/terramind/) and [HF model card](https://huggingface.co/ibm-esa-geospatial/TerraMind-1.0-large).
- **TiM is not a generic “fill missing JSON fields” tool:** it operates under documented **modality**, **band completeness**, and **`tim_modalities`** constraints.
- **Artifacts for Hub / cache:** Persist **schema-defined** rows for **every `tim_modality` the run imagined** — logits / class names / small raster thumbs / **`Coordinates` → WGS84** as separate capped fields inside **`tim_modality_outputs`** (not a single opaque blob). **Do not** store unbounded raw **H×W×C** float patches on Hub by default; pool or thumbnail per **`rules/12`**. Document **`model_version`**, **`merge_method`**, **`tim_modalities`**, and **standardization** in each row or sidecar manifest. **`Coordinates`** outputs are the **canonical optional source** for **cached AI-guess** lat/lon when product enables that modality. See **`12-python-gradio-terramind-server.md`** and **`plans/2026-04-07-gradio-terramind-backend.md`**.

## TerraMind — **full generation** (`FULL_MODEL_REGISTRY`, `*_generate`)

- Use **`FULL_MODEL_REGISTRY.build('terramind_v1_*_generate', …)`** with explicit **`modalities`** (inputs) and **`output_modalities`** per the [TerraMind guide — Generation](https://terrastackai.github.io/terratorch/stable/guide/terramind/). Prefer **`standardize=True`** unless you apply the documented pre-training scaling tables yourself.
- **IBM constraint (generation):** the guide states generation models **require all pre-trained bands for the inputs**—same “no cheating on band completeness” discipline as TiM, but applied to the **`_generate`** path rather than the frozen TiM inner generator.
- **Latency / cost:** `*_generate` uses **shared diffusion decoders**; **tiny/small are not much faster than base/large**—size mainly affects **quality** ([guide — Model Versions](https://terrastackai.github.io/terratorch/stable/guide/terramind/)). Expose **timeouts**, **503**, and **async job IDs** for long runs; **never** block ranked **guess submit** on generation completion (`rules/05-networking-leaderboard.md`).
- **Outputs to clients or Hub:** **schema-bound** (paths, MIME types, `model_version`, `output_modalities`, byte caps)—same trust hygiene as TiM and VLM (`rules/05-networking-leaderboard.md`, `docs/POI-PACKAGES-AND-OFFICIAL-CLIENTS.md` §4).
- **Reference notebooks (IBM):** [terramind_generation.ipynb](https://github.com/IBM/terramind/blob/main/notebooks/terramind_generation.ipynb), [any_to_any_generation.ipynb](https://github.com/IBM/terramind/blob/main/notebooks/terramind_any_to_any_generation.ipynb), [large_tile_generation.ipynb](https://github.com/IBM/terramind/blob/main/notebooks/large_tile_generation.ipynb) — behavioral reference for demos and batch Jobs, not shipped inside KMP.
